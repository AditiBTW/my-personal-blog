Modifying event field configuration value in event-driven microservices architecture

In this blog post, I'll be sharing my experience of trying to modify event field config value being passed through events in event-driven microservices architecture.
Context:
To understand our use case, let's assume we have many microservices (Service-A, Service-B ...) which consumes events generated by our microservice (Sevice-I).

Every event has a defined structure, along with pre-defined values associated with it's fields.

Now, let's assume we have one such event EventAlpha having a field key_alpha which can have any one of configured values: ALPHA1, ALPHA2 ... We discovered that the dev team woking on microservice (Service-E) has checked the configuration value for key_alpha to Alpha23. According to naming conventions of clean code practises, the value for key_alpha should be ALPHA23. Hence, we need to refactor Alpha23 to ALPHA23 for maintaining better consistency, clarity and convenience while writing code.

An event structure is considered as a contract between one microservice producing the event and other microservices consuming that event.
Hence, changing field values configuration can give rise to undesirable consequences in microservices consuming that event.

In our analysis, we identified 3 microservices (Service-A, Service-B, Service-C) directly consuming EventAlpha produced from our microservice (Service-I) and 1 microservice (Service-A) directly consuming key_alpha value of EventAlpha. Also we did not find any logic in Service-B and Service-C that used value of field key_alpha of EventAlpha, so we changed the configuration value to ALPHA23 and made respective changes in Service-A.

After pushing the code changes, we got many discrepancies arising due to change the configuration value to ALPHA23. On analysing the discrepancies we found:
  • Other teams working on Service-B and Service-C had merged their feature changes in respective repos. The newly added changes had logic which triggered different events based on the value received in key_alpha of EventAlpha.
  • Another microservice (Service-D) that was created to record and trace events which are consumed across various microservices (Service-A, Service-B ...) had now started consuming key_alpha field values to analyse for occurance of different configuration values.

On further analysis,
  • We found that the new set of events triggered using values of key_alpha in Service-B, Service-C and Service-D were then consumed by another set of microservices(Service-J, Service-E, Service-K).

We have to now consider one of the following approaches to resolve the discrepancy:
  • Revert the code changes related to configuration values of key_alpha.
  • Propagate the correction code related to configuration changes of key_alpha across all directly and indirectly dependent microservices.

The second approach is extremely difficult, time-consuming and cause blockage for dev teams working on respective services. Hence, we chose first approach and reverted our code changes for key_alpha value Alpha23.

The key learnings from this were:
  • Ensure proper code conventions are followed which introducing new configuration values for every event field.
  • When we need to modify an event field value, we need to throughly analyse all the microservices consuming that event field (directly/indirectly).
  • Also communicate with respective teams working on the dependent microservices to understand if any ongoing development uses that event field value.
