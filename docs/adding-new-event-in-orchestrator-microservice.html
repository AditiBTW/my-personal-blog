Adding new event in orchestrator microservice

Hi everyone, in this blog I'll be sharing how we added new event in our orchestrator microservice.
Let's assume we have a microservice [Service-1] which interacts with other microservices using events. Now Service-1 sends request to Service-2 to get some data, say record details for given user and when data is retrieved and sent to Service-1 successfully, Service-2 produces an event [Event-step-completed] which is consumed by Service-1.
Kafka is primarily used to stream the events that are produced and consumed within these microservices.
We found that Service-1 waits for response from Service-2 which impacted Service-1's performance and responsiveness to users. Also, depending on the load of requests on Service-2, the response time of request sent by Service-1 varied a lot.
We came up with a strategy to introduce a time-limit so that Service-1 waits for certain time within which is we get the response from Service-2, we will produce Event-step-completed from Service-2 otherwise Service-1 will produce Event-step-not-completed which is then consumed by respective consumer of Service-1.
We created the new event [Event-step-not-completed] using pre-existing event interface and added logic to produce Event-step-not-completed from Service-1 when response is not received from Service-2 within pre-defined time-limit. Next, we checked for the all the topics consumed by the consumers in Service-1 and assigned event [Event-step-not-completed] to the topic [Topic-step-not-completed]. We added logic in the consumer of Service-1 to consume event [Event-step-not-completed] and handle further processes for these users.
After creating the event [Event-step-not-completed] for the Service-1, we have
  • Improved the response time for Service-1 as we only wait for response from Service-2 for pre-defined time-limit after which we produce Event-not-completed.
After analysing, we found that we can further enhance our orchestrator microservice by
  • Implementing event [Event-step-not-completed] for other steps in our microservice [Service-1] and create a check-list of completed steps, which can be used to identify the step where our orchestrator microservice [Service-1] fails.
  • Adding a new functionality to resume any process in Service-1 from the last step which is completed in the check-list.
